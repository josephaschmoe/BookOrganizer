# Frontend Refactor Contract Freeze

This document freezes the production-facing frontend contracts for the phased `index.html` / `share.html` decomposition work. Until the decomposition is complete and explicitly re-approved, these contracts must remain unchanged.

Current baseline:
- Git commit observed before this refactor slice: `f0b6714`
- Frontend entrypoints: `public/index.html`, `public/share.html`
- Hosting rewrite: `/share/**` -> `public/share.html`

## Non-Negotiable Rules

- Do not change Firestore collection or document paths.
- Do not change document field names or expected shapes.
- Do not change Firebase Storage paths.
- Do not change callable function names or request/response contracts.
- Do not change existing public share token behavior or URL format.
- Do not rename or remove existing localStorage keys used by the app.
- Do not weaken or remove `_booksOwnedByUid` cross-user write guards.
- Default this refactor to `hosting`-only.
- Approved exception for the current slice: UI changes that depend on `restoreShareLink` or `copyCurrentCoverToBookPhoto` require a compatible Functions deploy in the same release.

## Share URL Compatibility

- Firebase Hosting must continue rewriting `/share/**` requests to `public/share.html`.
- Existing shelf and book share tokens must keep resolving without rotation.
- The public share page must continue supporting both shelf and single-book payloads through the current callable fallback behavior.

## Deployment Notes For This Refactor

- Pure file moves, CSS extraction, and script-load reorganization can still deploy with `firebase deploy --only hosting`.
- Any release containing UI that calls `restoreShareLink` or `copyCurrentCoverToBookPhoto` must deploy `hosting` and `functions` together.
- Do not ship the frontend half of those features ahead of the Functions export, or the UI will call missing callables in production.
- Continue treating any additional backend contract changes as out of scope unless they are explicitly reviewed and added here first.

## Allowed Refactor Scope

Safe changes in the decomposition project:
- moving inline CSS and JS into static asset files
- extracting shared read-only utilities
- reorganizing script load order without changing runtime behavior
- reducing duplication between `index.html` and `share.html` at the utility layer
- limited companion Functions changes only when required to preserve existing UX during decomposition, and only when documented in this file first

Out of scope for the safe decomposition phases:
- schema changes
- path migrations
- unapproved callable contract changes
- auth behavior changes
- share token rewrites
- frontend architecture rewrites that intentionally change event wiring or state ownership
